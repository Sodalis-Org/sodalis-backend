# Sécurité — Sodalis

Ce document répond au critère C2.2.3 : *« Les mesures prises permettent de couvrir les 10 failles de sécurité principales décrites par l'OWASP. »* Chaque ligne du tableau ci-dessous renvoie à un fichier réel du dépôt et à un test qui l'exerce — ce n'est pas une check-list déclarative.

Périmètre : `sodalis-backend` (4 services : `api-gateway`, `service-domus`, `service-labor`, `service-concordia`) et, pour A02, le frontend `sodalis-frontend` dans la mesure où il détermine où vit le jeton d'authentification.

## Risques acceptés

- **gRPC `service-domus` ↔ `service-labor` en `createInsecure()`, pub/sub Redis `sodalis_events` non signé.** Ni le canal gRPC (`VerifyUser`, `CreateTask`) ni les événements Redis ne portent d'authentification applicative : la protection repose entièrement sur l'isolation du réseau Docker interne (ces ports ne sont jamais publiés hors du réseau `sodalis`, cf. `docker-compose.prod.yml`). Ce choix est cohérent avec la décision d'architecture du projet : aucun hébergement public, cible de déploiement = hôte Docker unique. Ajouter du mTLS ou une signature HMAC sur ces canaux internes n'apporterait pas de bénéfice proportionné à la complexité pour ce périmètre — à réévaluer si l'architecture évoluait vers plusieurs hôtes ou un réseau partagé avec d'autres charges de travail.
- **`graphql-query-complexity` est estimé sans tenir compte de tous les arguments dynamiques.** L'estimateur utilisé (`simpleEstimator`) attribue un coût fixe par champ ; il ne module pas le coût en fonction d'un argument `limit`/`page` (aucun champ du schéma n'en expose qui influerait significativement sur le coût réel). Un raffinement par estimateur dédié serait à envisager si de tels champs apparaissaient.
- **Pas de CAPTCHA ni de vérification d'email à l'inscription.** Le rate limiting sur `/auth` (10 req/15 min par IP) limite l'automatisation à petite échelle ; une protection plus forte contre la création de comptes en masse n'a pas été jugée proportionnée à une application de gestion de colocation à usage fermé (les comptes se créent via un code d'invitation de colocation).
- **`secure: true` sur le cookie d'authentification dépend de `NODE_ENV=production` et donc d'une terminaison TLS en amont**, non traitée par le code applicatif (reverse proxy / terminaison TLS relève du protocole de déploiement, documenté dans `DEPLOYMENT.md`).
